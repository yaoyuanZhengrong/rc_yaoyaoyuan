# API 通知系统 (Notification Service)

本项目是针对 “AI Coding 作业：API 通知系统设计与实现” 的 MVP (最小可行性产品) 实现。

## 1. 问题理解与设计理念

本系统的核心需求是：**为内部多个业务系统提供一个可靠、解耦的外部 API 通知投递通道**。

### 核心痛点
* **外部 API 不稳定**：供应商 API 可能超时、宕机或限流。
* **业务系统耦合**：业务系统不应同步等待外部 API 返回，也不应处理复杂的重试逻辑。
* **格式多样性**：不同供应商对 Header 和 Body 的要求各异。

### 设计原则
* **异步化**：通过消息队列（Kafka）实现削峰填谷，解耦生产与消费。
* **持久化**：所有通知请求必须存入数据库（MySQL），确保可审计、可追溯。
* **可靠性**：支持“至少一次”投递语义，内置指数退避重试逻辑。

---

## 2. 整体架构与核心设计

系统采用典型的 **API + MQ + Worker** 架构：

1.  **API Interface (FastAPI)**: 接收业务系统请求，进行参数校验，并将消息发送至 Kafka。
2.  **Kafka (Message Queue)**: 充当消息中转站，确保消息不丢失并提供缓冲。
3.  **Worker (Consumer)**: 
    *   从 Kafka 读取消息。
    *   根据 `msg_channel` 匹配 `api_configs` 中的供应商配置。
    *   执行实际的 HTTP(S) 请求。
    *   根据返回结果更新数据库状态或触发重试。
4.  **MySQL (Storage)**:
    *   `api_configs`: 存储供应商的 URL、Header 模板和 Payload 模板。
    *   `notification_records`: 记录每条通知的生命周期、状态和错误信息。

## 3. 关键工程决策与回答 (Q&A)

### 1. 系统边界
* **解决的问题**：
    * **可靠投递**：通过 Kafka + MySQL 状态机 + 定时扫描器，确保即使在极端情况下（如 Worker 全体宕机后重启）消息也能最终送达。
    * **协议适配**：支持通过模板化配置（ApiConfig）动态适配不同供应商的 API 格式，无需修改代码即可增加新渠道。
    * **流量控制**：通过 Kafka 消费者的并发数控制，保护外部供应商不被瞬间高并发冲垮。
* **不解决的问题**：
    * **业务逻辑校验**：本系统不校验业务参数的正确性（如订单号是否存在），仅负责投递。
    * **消息顺序性**：通知类消息通常对顺序不敏感。为了追求高并发处理，我们不保证绝对顺序（除非配置 Kafka 的 Partition Key）。
    * **供应商 API 性能**：我们无法加速外部 API，只能通过异步化避免其影响内部业务。

### 2. 可靠性与失败处理
* **投递语义**：采用 **“至少一次 (At-least-once)”**。
    * 流程：Worker 处理完消息并成功收到外部 2xx 响应后，才会提交 Kafka Offset 并更新 DB 状态。
    * 幂等性：建议业务系统提供唯一的 `business_id`，Worker 在执行前会检查 DB，防止重复投递。
* **失败策略**：
    * **指数退避重试**：默认支持 5 次重试，间隔分别为 10s, 1min, 5min, 10min, 30min。
    * **死信队列 (DLQ)**：超过最大重试次数的消息进入 `FAILED` 状态，并记录详细错误原因，由人工介入或后续自动化脚本处理。

### 3. 取舍与演进
* **取舍**：
    * **为什么选 Kafka 而不是 Redis 队列？** 虽然 Redis 更轻量，但 Kafka 提供了更好的持久化保障和消息回溯能力，更符合“通知系统”对可靠性的极致追求。
    * **为什么需要 MySQL 记录状态？** 纯 MQ 难以进行复杂的任务审计（例如：查询某个用户过去一小时的所有通知）。DB 提供了上帝视角。
* **演进路径**：
    * **第一版 (MVP)**：单 Topic 处理所有通知。
    * **高并发演进**：根据 `msg_channel` 拆分 Kafka Topic，实现优先级队列（例如：验证码 Topic 优先级高于营销通知 Topic）。
    * **监控与告警**：引入 Prometheus + Grafana 监控投递成功率、延迟情况，并在供应商 API 连续失败时触发告警。

## 4. 如何运行

1.  安装依赖：`pip install -r requirements.txt`
2.  配置 `.env`: 设置 MySQL 和 Kafka 连接信息。
3.  启动 API：`python http_server.py`
4.  启动消费者：`python kafka_consumer.py`
