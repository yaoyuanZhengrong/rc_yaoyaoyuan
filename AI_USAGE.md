# AI 使用说明 (AI Usage Report)

本项目是在 AI 的深度辅助下完成的，涵盖了从需求分析、架构设计到代码实现的全过程。以下是关于 AI 使用情况的详细说明：

## 1. AI 提供的关键帮助

*   **架构选型与论证**：AI 建议采用 **Kafka + MySQL** 的组合，并清晰地解释了这一选型在“可靠性”与“审计追溯”方面的优势，帮助我快速锁定了技术栈。
*   **代码框架生成**：AI 快速生成了基于 FastAPI 的 HTTP Server、基于 Kafka-python 的生产/消费逻辑，以及 SQLAlchemy 的模型定义，节省了大量编写样板代码的时间。
*   **重试算法实现**：AI 提供了完善的**指数退避重试**算法建议，并帮助实现了基于 `available_after` 时间戳的定时扫描重试机制，这是系统可靠性的核心。
*   **设计模式应用**：在消息投递环节，AI 建议并实现了 **Strategy (策略模式)**，使得系统能够轻松扩展不同的通知渠道（SMS, Email, Push 等）而无需修改核心业务流程。
*   **引入 `ApiConfig` 动态模板**：AI 建议引入 `ApiConfig` 表，通过 JSON 模板来定义 Header 和 Payload，以应对供应商接口格式的频繁变动。
*   **原子化状态更新 (State Machine)**：AI 建议在 Worker 更新 DB 状态时使用 `from_status` 进行校验，实现一个简单的状态机，防止并发环境下的“脏写”或状态回退。
*   **定时扫描器的双重校验**：在扫描 `RETRY_PENDING` 记录时，AI 建议使用 `with_for_update(skip_locked=True)`，防止分布式部署时重复投递。

## 2. 未采纳的 AI 建议

*   **过度工程化的分表建议**：AI 曾建议在第一版就引入 MySQL 分库分表来处理海量通知记录。
    *   **未采纳理由**：作为 MVP 版本，应优先保证逻辑的正确性和快速交付。当前的单表索引已能支撑初期流量，过早引入分表会增加运维复杂度和开发成本。
*   **分布式事务 (2PC/TCC)**：AI 曾探讨过在 Kafka 写入和 MySQL 记录之间使用分布式事务来保证强一致性。
    *   **未采纳理由**：在通知系统中，通过“先写 Kafka，消费端再写 DB”的最终一致性模型已经足够。分布式事务会极大地降低吞吐量并增加系统脆弱性。

## 3. 人工（我）做出的关键决策

*   **`message_sender` 策略模式封装**：我决定为不同的消息提供商封装对应的参数校验和消息发送逻辑，使用了策略模式。
    *   **原因**：这保证了系统极强的扩展性，未来接入新的供应商只需增加新的策略类，而无需修改核心业务流程。
*   **深思熟虑的重试机制**：我反复思考并设计了消息失败重试部分的逻辑和实现。
    *   **原因**：重试是分布式系统可靠性的基石。我不仅实现了指数退避，还考虑了状态回退保护和扫描冲突，确保重试过程既能最终成功，又不会造成系统过载或数据错乱。
*   **极致的高可用网关设计**：在设计消息投送网关时，我决定只进行参数校验和 Kafka 消息投递，不做数据库入库操作。
    *   **原因**：为了保证网关层极强的服务可用性和极低的响应延迟。入库操作交由下游消费端异步完成，避免了数据库成为网关吞吐量的瓶颈，也降低了因数据库抖动导致网关不可用的风险。

---

## 总结

AI 是一个极强的“百科全书”和“代码生成器”，它能提供标准化的最佳实践。但**工程决策的核心在于“取舍”**。我作为人类开发者，更多地是在平衡系统的**复杂度、开发效率与未来的可扩展性**，确保系统在满足当前需求的前提下，保持最简洁、最稳健的状态。
